Cockpit Architecture Multi-agent orchestration layer — system and module reference

Date: 2026-05-28 Branch: develop Commit: 75c8d38 767 files 91 processes 6,011 edges

1. What Cockpit Is

Cockpit is a multi-agent orchestration layer for managing AI coding agents across multiple software projects. It is not a Claude Code tool — Claude Code is the reference implementation; the system supports Codex, OpenCode, and Gemini CLI through a pluggable driver abstraction.

Conceptually, Cockpit wraps a terminal multiplexer (cmux) with a role-based delegation system: a Command session dispatches work to Captain sessions (one per project), which delegate coding tasks to ephemeral Crew sessions running in isolated git worktrees.

2. Role Hierarchy

Cockpit defines three roles that form a strict hierarchy. Each role maps to a distinct agent session type, with its own template file in plugin/templates/ and its own configuration in ~/.config/cockpit/.

Command — Orchestration Overseer

Template: plugin/templates/command.claude.md
Spawned by: cockpit command --task <briefing|learnings-review|wiki-aggregate>
Skill: cockpit:command-ops

The Command is the top-level dispatcher. It is spawned on-demand for a single task (no persistent Command session). Its sole job is to delegate work to project Captains and report status to the user. It may never read, write, or search project source code. It has access to ~/.config/cockpit/config.json, the hub vault, and cockpit CLI/ cmux commands.

Captain — Project Leader

Templates: plugin/templates/captain.claude.md (Claude), plugin/templates/captain.generic.md (non-Claude agents)
Spawned by: cockpit launch <project>
Skill: cockpit:captain-ops

One Captain per registered project. The Captain is a coordinator, never a coder. Its responsibilities:

  • Spawn Crew sessions via cockpit crew spawn for every coding task
  • Send follow-up turns to existing Crews via cockpit crew send
  • Inspect Crew screens via cockpit crew read
  • Review and merge completed Crew work
  • Record learnings; write handoff files at session end
  • Apply Karpathy principles during code review

Crew — Worker Context

Templates: plugin/templates/crew.claude.md (Claude), plugin/templates/crew.generic.md (generic), plugin/templates/crew.opencode.md (OpenCode)
Spawned by: cockpit crew spawn <project> "<task>" [--agent <agent>]
Lifecycle: spawned as a cmux tab, given a task prompt, operates in a git worktree, signals completion via cockpit crew signal done.

Crews are ephemeral, single-session agents. Each Crew runs in a git worktree (branch isolated from main). A Crew must commit its work and send an explicit cockpit crew signal done before exiting. If blocked, it signals crew signal blocked; if unrecoverable, crew signal failed. All signals are routed through the daemon state machine (see Control Plane).

Crews may be backed by any agent type (--agent claude|codex|opencode|gemini). The crew.generic.md template provides the same rules for non-Claude agents. Claude crews additionally receive per-crew settings.local.json with Stop/SubagentStop/PostToolUse hooks wired for liveness and state tracking.

For complex multi-step tasks, Claude Crews may use GSD (Global Step Dispatch) wave-based execution (/gsd:plan-phase, /gsd:execute-phase) which spawns subagents per step with fresh 200K-token contexts.

Rule: Captains never code. Captains spawn Crews for every coding task. Even a one-line fix gets a Crew session. This is enforced in the Captain templates.

3. Three Pluggable Slots

The three plugin slots (runtime, workspace, notifier) follow an identical registry pattern. Each slot has:

  • types.ts — driver interface definition
  • registry.ts — factory registry with per-project resolution
  • index.ts — barrel exports
  • One or more concrete implementations as factory functions

The configuration in ~/.config/cockpit/config.json selects which driver to use globally (config.runtime, config.workspace, config.notifier) or per-project (config.projects[].runtime, etc.).

SlotInterfaceMethodsImplementationsDefault
Runtime RuntimeDriver probe, list, spawn, send, readScreen, stop, newPane, closePane, spawnInjector, listSurfaces cmux cmux
Workspace WorkspaceDriver probe, read, write, exists, list, mkdir obsidian (local fs) obsidian
Notifier NotifierDriver probe, notify cmux (routes to Command workspace) cmux

Runtime Driver (@cockpit/workspaces)

The runtime abstracts the terminal multiplexer. Today only cmux is implemented (packages/workspaces/runtimes/cmux.ts). It wraps the cmux binary using execFileSync with argv arrays (no shell injection). The RuntimeRegistry resolves which driver to use per-project from config.

The runtime is the thickest plugin interface: it manages workspaces (captain sessions), panes/tabs (crew sessions), injector processes (mailbox tailers), and surface listing. The cmux driver implements all 10 interface methods.

Workspace Driver (@cockpit/workspaces)

The workspace abstracts persistent storage for cockpit's knowledge vaults (wiki, learnings, daily logs, handoffs). Today only Obsidian is implemented, which is a local filesystem driver with path-escape protection. The WorkspaceRegistry resolves both hub() (from config.hubVault) and forProject() (from config.projects[].spokeVault) drivers. The factory pattern allows future providers (e.g., Obsidian Remote, Notion API, S3).

Notifier Driver (@cockpit/workspaces)

The notifier is the simplest plugin slot: a single notify(message) call. Today only cmux is implemented, which routes notifications through the cockpit runtime send --command CLI back to the Command workspace.

4. Agent Drivers

Agent drivers (packages/agents/drivers/) abstract the coding agent CLI. Each driver is a factory function returning an AgentDriver interface with four methods: probe, buildCommand, parseOutput, and stop. The CapabilityRegistry holds all registered drivers, probes each for installed version/capabilities, and validates agent suitability per role.

DriverFileInteractive modeHeadless modeAuto-approve
Claude packages/agents/drivers/claude.ts claude (bare TUI, no -p) claude -p "..." --print --dangerously-skip-permissions
Codex packages/agents/drivers/codex.ts codex (via daemon AppServer) codex exec "..." --json --full-auto --full-auto
OpenCode packages/agents/drivers/opencode.ts opencode (bare TUI) opencode run "..." via per-crew opencode.json config (all-permissions)
Gemini packages/agents/drivers/gemini.ts generates Gemini CLI interactive command via prompt flag via driver config

Capability-Based Routing

Each agent driver exposes a set of AgentCapability values: teams, json_output, sandbox, model_routing, skills, auto_approve, streaming, prompt_file. The ROLE_REQUIREMENTS map in packages/agents/drivers/types.ts:59 defines which capabilities each role requires:

  • All roles require auto_approve
  • Command additionally prefers teams and json_output
  • Captain prefers teams, model_routing, skills
  • Crew prefers json_output, sandbox

5. Control Plane (cockpitd Daemon)

The control plane daemon (packages/core/, 14 files) is a macOS launchd-managed background process that supervises all crew execution. It provides a task state machine, watchdog/heartbeat, push notification mailbox, interactive hook bridge, and Codex interactive driver.

The daemon is defined in packages/core/launchd.ts as a launchd agent (com.cockpit.daemon) with KeepAlive + ThrottleInterval (10s) to prevent crash loops. ensureDaemon() writes the plist and bootstraps the service. On startup (packages/cli/index.ts:47), ensureDaemon() runs before any command.

The daemon listens on a Unix domain socket at ~/.config/cockpit/cockpit.sock using a JSON-line protocol (packages/core/protocol.ts). Communication includes request/response RPC and long-lived attach streaming connections for interactive Codex sessions.

Task State Machine

The state machine (packages/core/state-machine.ts) is a pure reducer function: reduce(record: TaskRecord, event: ControlEvent, now: number) => TaskRecord. It is fully testable without I/O. The daemon (packages/core/daemon.ts) injects side effects (store, clock, process launcher, notifier).

Key state machine properties (packages/core/state-machine.ts):

  • Terminal states (TERMINAL_STATES): only done and failed are absorbing — any late/duplicate event is ignored.
  • Recoverable stalls: stalled is not terminal. The watchdog can recover a stalled task to working if a heartbeat arrives.
  • awaiting-input: the PostToolUse-like state that separates turn completion from task completion (anti-#2576 invariant).
  • Pure function: the reducer never mutates input; it always returns a new record (spread + targeted overrides).

Watchdog / Heartbeat

The watchdog (packages/core/watchdog.ts) implements stall detection as a pure function:

  • evaluateStall(rec, now) — if a working task has not received a heartbeat within heartbeatBudgetMs, produces a stalled record.
  • recoverStall(rec, now) — produces a working record from a stalled record (caller must verify heartbeat freshness).

The daemon sweeps all tasks on a 30-second interval (packages/cli/packages/core/cockpitd.ts:287-290). Interactive tasks (Claude, OpenCode, Codex) receive heartbeats from the PostToolUse hook (Claude), from the daemon's state ledger (OpenCode), or from the AppServer event stream (Codex).

Hook Bridge (Interactive Claude Crews)

For Claude interactive crews, the daemon does NOT own the process. The crew runs in a cmux tab. The daemon's role is purely the state ledger:

  • On dispatch, emit task.started to transition submitted → working
  • On each Claude turn end, the Stop hook fires (embedded in per-crew settings.local.json), emitting task.turn.completed → awaiting-input
  • On the next Turn, PostToolUse emits a heartbeat → back to working
  • On actual task completion, cockpit crew signal done emits task.done → done

The hooks are injected per-crew via packages/cli/lib/per-crew-settings.ts: writePerCrewSettings() writes a --settings-compatible JSON file with Stop/SubagentStop/SessionEnd hooks merged. writePerCrewSettingsLocal() writes .claude/settings.local.json at the project level (auto-loaded, not clobbered by cmux wrapper settings).

Codex Interactive Driver

Codex interactive crews use a fundamentally different architecture: the daemon owns the Codex child process via its CodexInteractiveDriver (packages/agents/src/codex/), which manages a single AppServerClient instance. Each task maps to a thread inside that child.

The Codex architecture supports:

  • Attach streaming — cmux clients can attach to a Codex task and receive real-time delta frames via the Unix socket.
  • Gate promotion — when Codex requests approval or input and no client is attached, a 5-second timer promotes the request to a persistent Gate in the store. A later-attaching client can see and resolve the gate.
  • Inbound operations — attached clients can say, steer, interrupt, and answer on the Codex task.
  • Restart-reattach — on daemon restart, non-terminal interactive Codex tasks with a resumeRef are automatically reattached.

6. Cross-Agent Projection Layer

The projection layer (packages/agents/projection/, 9 files) implements issue #31: synchronizing cockpit's instruction templates into each agent's native configuration format.

Each supported agent type has an emitter that knows where to write its configuration files (user-level and project-level):

TargetFileDestinationFormat
Cursorpackages/agents/projection/cursor.ts.cursor/rules/*.mdc.mdc rule format
Codexpackages/agents/projection/codex.tsAGENTS.mdMarkdown (marker merge)
Gemini CLIpackages/agents/projection/gemini.tsGEMINI.mdMarkdown (marker merge)
OpenCodepackages/agents/projection/opencode.tsAGENTS.mdMarkdown (marker merge)

Note: Claude is not a projection target. CLAUDE.md is cockpit's canonical/source instruction format that Claude Code reads natively — other agents are projected from it, not to it.

All emitters use a marker-based merge strategy (packages/agents/projection/marker.ts): content is inserted between <!-- cockpit:start --> and <!-- cockpit:end --> markers. This allows surgical updates without clobbering user content outside the markers.

The ProjectionRegistry (packages/agents/projection/registry.ts) holds emitter factories and creates the correct emitter by name. The cockpit projection CLI command (packages/cli/commands/projection.ts) drives this: it reads user-level source (plugin/templates/ templates + plugin/skills/) and project-level source (project AGENTS.md), then emits to all configured targets via ProjectionEmitter.emit().

Direction per multi-agent spec (2026-04-24): AGENTS.md is the canonical instruction format. CLAUDE.md becomes a thin wrapper. Skills stay portable markdown — Claude reads via Skill tool, others via AGENTS.md inclusion.

Configuration (packages/shared/config.ts)

The CockpitConfig interface (packages/shared/config.ts:110-134) defines the top-level configuration shape:

  • commandName — workspace name for the command session
  • hubVault — path to the cross-project hub vault (~/cockpit-hub by default)
  • projects — Record<string, ProjectConfig> mapping project names to their path, captain name, spoke vault, host, group, runtime, workspace, and notifier overrides
  • agents — registered agent CLIs and their driver mappings
  • defaults — maxCrew, worktreeDir, teammateMode, permissions (auto-approve per role), models (model routing per role), roles (agent + model per role)
  • runtime / workspace / notifier — default plugin slot selections
  • projection — projection target list
  • metrics — flag + metrics file path

Dashboard (@cockpit/web)

Three files implementing the terminal dashboard:

  • read-status.ts — queries each project's task state from the daemon via the {kind:"list"} protocol message, returning live per-task status
  • render.ts — color-coded terminal rendering with state icons
  • sync-hub.ts — mirrors project status to hub vault at {hubVault}/projects/{project}.md

Library (packages/cli/lib/)

Seven utility files:

  • runtime-sync.ts — mirrors plugin/, scripts/, plugin/templates/ from source to ~/.config/cockpit/
  • cmux-bin.ts — resolves cmux binary path (env → config → PATH → fallback)
  • per-crew-settings.ts — writes per-crew Claude settings and OpenCode config
  • canonical-source.ts — reads skills and AGENTS.md for projection
  • vault-layout.ts — ensures spoke vault directory structure
  • daily-logs.ts — read/write daily log files

Entry Point (packages/cli/index.ts)

The CLI entrypoint (packages/cli/index.ts) does two things before registering any command:

  • Calls ensureRuntimeSynced() — idempotent mirror of source-managed dirs to runtime (~/.config/cockpit/), so skills, role templates, and scripts can never silently drift.
  • Calls ensureDaemon() — idempotent launchd plist management, ensuring the control-plane daemon is running.

Both are best-effort and never throw — the CLI fails loud later if the socket is unreachable.
