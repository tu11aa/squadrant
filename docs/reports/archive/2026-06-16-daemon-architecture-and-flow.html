1 · The pieces

The daemon (cockpitd) is the brain — a long-lived launchd service that owns the task ledger + mailbox. cmux hosts every crew and the captain as surfaces.

CrewCaptain DaemonMailbox Relaycmux

2 · A crew finishes → daemon writes the mailbox

cockpit crew signal done hits the daemon socket. The daemon does two things: updates the ledger (state → done) and appends one entry with the next monotonic seq.

3 · Delivery — two interchangeable modes

A mailbox entry isn't a notification until something reads the cursor and pushes it to the captain's pane. Exactly ONE of these runs at a time (both = double-delivery).

A · Relay  DEFAULT

  • notify-relay runs inside the cmux tree (a cmux-descended helper).
  • Tails the mailbox from the captain cursor every ~1s (drain()).
  • Each new entry → cmux send to the captain surface → advance cursor.
  • Works out-of-the-box: cmux trusts its own descendants.
  • Zero setup — captain runs cockpit relay supervise.

B · Daemon-direct  OPT-IN · #332

  • The daemon itself reads the cursor & calls cmux send — no relay tab.
  • Gated by config daemonDirectCmux: true.
  • Same CaptainDelivery module as the relay → identical behaviour.
  • Requires cmux socketControlMode: automation (slide 5) — the daemon is not a cmux descendant.
  • Re-entrancy + stale-skip guards mirror the relay's drain().
HYBRID goal: probe cmux → use B if configured, else fall back to A. The relay is never deleted.

4 · The full path of one notification

A ● packet travels crew → daemon → mailbox → delivery → captain. Top lane = relay (default), bottom lane = daemon-direct.

relay lane (default)daemon-direct lane (opt-in)cursor advances only after a successful send → in-order, at-least-once.

5 · Why daemon-direct needs a cmux switch

cmux's control socket checks the caller's process ancestry at accept(), governed by automation.socketControlMode. This gate is the entire reason the relay exists.

Set via ~/.config/cmux/cmux.json (file-managed, JSONC) + one cmux restart. Same-user threat model → no password needed.

6 · Mailbox mechanics & the storm guards

The mailbox is dead simple — an append-only log + a read cursor — but delivery needs four safeguards (the relay had them; daemon-direct had to learn them, via two storms 🌩).

The data

  • <project>.log — append-only; each line {seq, ts, taskId, name, kind}; rotated to .log.1/.2. Nothing is ever deleted → full audit trail.
  • <project>.captain.cursor — {lastAckedSeq}. Read from lastAcked+1; advance only after a successful send.
  • Empty / corrupt cursor ⇒ treated as "start fresh" (never throws).

The 4 delivery guards

  • defer-while-typing — hold a send while the captain is mid-keystroke (#258/#302).
  • stale-skip — silently ack entries older than session start, so a reset cursor never re-floods the backlog.
  • re-entrancy guard — a slow tick must not overlap the next (else the same entry delivers twice → storm).
  • atomic cursor write — unique tmp + rename; concurrent writes never corrupt it.
Both modes share one CaptainDelivery module → relay and daemon-direct behave identically. Flip the flag, behaviour is unchanged.
