cockpit — /src & /control relation graphs

Generated 2026-06-17 from the live import graph (relative imports only; vendored codex/protocol/v2 & tests excluded). Companion to the architecture decks for the cockpitd refactor decision.

TL;DR for the refactor. cockpitd.ts (1008 LOC) is a god composition-root — it imports 16 of the control-plane modules and wires every I/O seam (server, attach, delivery, probes, drivers, gates). daemon.ts (503 LOC) is a clean core — 5 imports, all pure-logic. The split proposed in the handoff (cockpitd.ts → src/control/daemon/{server,attach,delivery,probes,drivers,gates}.ts via a thin DaemonContext) is justified by the fan-out below: each cockpitd import cluster maps cleanly to one proposed module. daemon.ts logic stays as-is.

Reading it. commands/ is the CLI hub — every cockpit subcommand lives here and reaches into control/ (28 edges) to talk to the daemon. control/ is a near-leaf: it depends only on runtimes/ (to launch agent sessions) and config. That low outward coupling is good — the control plane doesn't reach back up into the CLI. The weight is all inside control/, which §2 unpacks.

3 · cockpitd.ts fan-out → proposed split

Each cockpitd import cluster maps to one proposed module under src/control/daemon/. daemon.ts (core) is untouched.

Proposed moduleAbsorbs (cockpitd imports today)Why grouped
daemon/server.tsprotocol.ts, socket accept loop, request routingWire protocol → handlers
daemon/attach.tsheadless-launcher.ts, crew-pane-reader.ts, liveness.tsSession attach + liveness
daemon/delivery.tsmailbox.ts, delivery/captain-delivery.ts, relay-healer.tsNotification delivery (relay + daemon-direct)
daemon/probes.tssnapshot.ts, cmux/events-bridge.ts, opencode/sse-bridge.tsLifecycle probes / event ingestion
daemon/drivers.tscmux/daemon-cmux.ts, codex/driver.tsPer-agent driver wiring
daemon/gates.tscodex/gate.ts, approval/permission gatesPermission/approval seams
cockpitd.ts (~150 LOC)builds DaemonContext, calls createDaemon(), mounts the 6 modulesThin composition root

Refactor risk read

  • watch types.ts has fan-in 15 — it's the shared vocabulary. Leave it as the single source of truth; the split should not fragment it.
  • safe daemon.ts is already isolated (5 pure imports, fan-in 2). The split is purely about decomposing the HOST file, not the core.
  • target All 16 edges to slice originate from one file — the blast radius of the split is contained to cockpitd.ts + its new submodules; callers of the daemon (via protocol.ts over the socket) are unaffected.
  • Optional: rename daemon.ts → core.ts to end the daemon/cockpitd naming confusion (cockpitd = HOST, daemon = CORE). Cosmetic; do it in the same PR or skip.
  • Sequencing: the handoff recommends this split before #333 (NativeHookSource), since #333 adds another probe seam that would otherwise land in the already-overloaded HOST file.
