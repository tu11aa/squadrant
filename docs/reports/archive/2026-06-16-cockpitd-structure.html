⚙ Cockpit — Daemon Structure & Split Proposal

core vs host
space / → · A = autoplay

1 · Two layers (the seam already exists)

cockpitd.ts is NOT the brain — it's the host process (I/O, sockets, timers, drivers). The brain is daemon.ts = createDaemon(), a near-pure core with all I/O injected. They talk through one function: daemon.handle(req).

The split is healthy in principle — but the host (left) has accreted ~10 concerns into one 1008-line function. That's the weight.

2 · daemon.ts — the core (createDaemon)

Near-pure. Owns task lifecycle + the "when do we notify the captain" policy. Depends only on small pure modules. This part is already clean — leave it.

core logicsmall pure depsNo sockets, no timers, no cmux — testable by injecting fakes. ~503 LOC, cohesive.

3 · cockpitd.ts — the host (startCockpitd) · the heavy one

One 1008-line function wires ~10 distinct concerns + 20 imports + 9 socket handlers + 4 timers. Everything below lives inside a single closure — that's why it feels heavy.

Each colour = a concern that could become its own module. They share little state — mostly the store, the daemon handle, and a few maps.

4 · Proposed split — shrink the host to a wiring file

Extract each concern into a focused module under src/control/daemon/. startCockpitd becomes a thin composition root: build deps → createDaemon → mount modules → start timers. Target: ~150 LOC.

Extract from cockpitd.ts →

  • daemon/server.ts — socket + the 9 handlers (request router) · ~rpc
  • daemon/attach.ts — attach fan-out / AttachFrame broadcast
  • daemon/delivery.ts — daemon-direct loop + captain discovery + close-reap
  • daemon/probes.ts — surface-liveness + blocked probe wiring
  • daemon/drivers.ts — codex / opencode / cmux-events bootstrap
  • daemon/gates.ts — gate promotion timers
  • delete relay-proxy queues (with #332 relay removal)

Leave alone (already cohesive)

  • daemon.ts — the core. Maybe rename → core.ts to end the daemon.ts/cockpitd.ts confusion.
  • state-machine.ts, watchdog.ts, liveness.ts, store.ts
  • mailbox.ts, delivery/captain-delivery.ts
  • Pre-req: a thin DaemonContext ({store, daemon, log, cmux, cfg}) passed to each mounted module so they don't re-close over startCockpitd's locals.
Result: 1008-LOC closure → ~150-LOC composition root + 6 focused modules (~80-150 LOC each). Each independently testable. Lower risk for #332/#333 work.
