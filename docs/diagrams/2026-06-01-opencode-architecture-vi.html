Opencode trong Cockpit — kiến trúc & vì sao "dễ" nhất

01/06/2026 · báo cáo vòng đời · nguồn: src/control/opencode/sse-bridge.ts (PR #188), src/commands/crew.ts, kiểm chứng trên opencode 1.15.12

Tóm tắt 1 dòng: Opencode "dễ" vì nó là agent duy nhất có CẢ HAI thứ tốt nhất của hai agent kia mà không dính nhược điểm nào: (1) chạy CLI thật không sandbox → signal hoạt động ngay (như claude), và (2) có HTTP + SSE event bus đáng tin → tự phát hiện idle/permission (như codex), qua một kết nối duy nhất.

1 · Opencode hoạt động thế nào (CLI thật + SSE bus)

Crew chạy CLI opencode thật trong tab cmux (như claude). Điểm đặc biệt: TUI opencode nhúng sẵn một HTTP server. Ta khởi chạy opencode --port N để server lắng nghe ở cổng biết trước, rồi daemon mở một kết nối SSE dài hạn tới GET 127.0.0.1:N/event và nhận mọi sự kiện bus.

2 · Vì sao "dễ" — 4 lý do

① Không sandbox → signal chạy ngay

Khác codex (bị Seatbelt workspace-write chặn socket), opencode chạy như tiến trình thường trong môi trường captain. Lệnh cockpit crew signal connect thẳng tới daemon — 4 checkpoint dùng signal (CP1/2/5/6) PASS mà không cần code gì thêm.

② Có event bus thật → tự phát hiện, không scrape

Khác claude (chỉ có hooks, lại chập chờn), opencode có opencode serve + SSE /event: một stream đầy đủ session.idle, message.*, permission.asked… Wire-format ổn định {type, properties}, một kết nối nhận hết. Không phải parse màn hình.

So với 2 agent kia: codex tốt nhờ giao thức nhưng dính sandbox; claude không sandbox nhưng tín hiệu (Stop) chập chờn. Opencode gỡ được cả hai nút thắt cùng lúc.

③ Sửa idle (CP4) là thay đổi rất nhỏ

State-machine vốn đã hỗ trợ task.turn.completed → awaiting-input (codex dùng sẵn). Nên đóng CP4 chỉ gồm: spawn kèm --port N + 1 module SSE map session.idle → task.turn.completed + nối dây cổng. Phần lõi state-machine gần như không đụng.

Đối lập hoàn toàn với claude: cùng tính năng "idle ping" mà claude phải qua #182→#183→#185→#185b rồi revert; opencode chỉ cần 1 cầu nối SSE + 1 guard nhỏ, chạy tin cậy ngay.

④ Hệ sinh thái sẵn sàng

OpenAPI 3.1 spec, SDK chính thức @opencode-ai/sdk với event.subscribe(), opencode attach, resume qua --session/--continue. Tích hợp dựa trên API công khai, không phải hack.

3 · Điểm mạnh / Điểm yếu — Opencode

✅ Điểm mạnh
  • Có cả hai đường: explicit signal (không sandbox) và tự phát hiện qua SSE — không agent nào khác có đủ cả hai.
  • Event bus đáng tin: 1 kết nối SSE = mọi sự kiện vòng đời (idle, permission, message).
  • Dễ tích hợp nhất: idle bridge là module nhỏ, state-machine sẵn sàng.
  • API/SDK/OpenAPI công khai; resume; permission.asked sẵn cho CP3 tương lai.
  • Idle tin cậy (không như Stop hook của claude).
❌ Điểm yếu
  • CP3 permission còn hoãn: config hiện auto-approve mọi tool → chưa có luồng captain-duyệt cho opencode.
  • TUI gốc không mở /event nếu thiếu --port (mặc định dùng unix socket ẩn) → phải cấp cổng tường minh.
  • API plugin/event có thể đổi giữa các bản 0.x (dự án tốc độ cao) → rủi ro bảo trì.
  • Watchdog stall vẫn tắt (budget 24h) — hiện dựa vào SSE cho liveness; nếu server chết im, cần xử lý reconnect.
  • Mặc định trong setup này chạy model free (DeepSeek) — chỉ là cấu hình, nhưng cần lưu ý chất lượng.
