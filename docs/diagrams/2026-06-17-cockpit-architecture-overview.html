1 · Cockpit in 5 layers

Cockpit is a multi-agent orchestration layer. Read top→down: the user types a CLI command → the CLI spawns role-sessions + talks to the control plane (the daemon) → the daemon drives agents & the terminal through adapters → everything persists to storage.

③ control plane = where daemon & cockpitd live — between the roles (above) and the adapters (below)

2 · One command travels through the stack

Example cockpit crew spawn …: CLI → daemon (writes the ledger) → cmux adapter (creates a surface, spawns the agent) → crew runs → on finish, daemon → mailbox → delivery → captain. daemon/cockpitd is the relay station in the middle.

For the "done → mailbox → captain" leg in detail, see the deck 2026-06-16-daemon-architecture-and-flow.html.

A map to locate any file in the project.

③ Control planesrc/control/ — cockpitd.ts, daemon.ts, store, mailbox, state-machine, watchdog, liveness, snapshot, protocol, codex/ opencode/ cmux/ interactive/ delivery/The brain. daemon = cockpitd (host) + createDaemon (core). Tracks task lifecycle, delivers notifications.
④ Adaptersdrivers/ (agents) · runtimes/ (cmux) · notifiers/ · workspaces/ · projection/Each = a registry + types = a plugin slot. Swap cmux→another driver, add an agent… without touching the control plane.
⑤ Persistence~/.config/cockpit/ · cockpit-hub/ · src/lib/config.json, state (ledger/mailbox/cursor), hub-spoke vault; lib = shared (cmux-bin…).
