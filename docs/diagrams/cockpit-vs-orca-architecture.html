claude-cockpit vs Orca

Two approaches to multi-agent orchestration — architecture diagrams & mapping

This page maps the real architecture of claude-cockpit (this repo, as of develop branch May 2026) against Orca (stablyai/orca, studied in docs/research/2026-05-19-orca-full-system-study.md). Every box in the cockpit diagrams corresponds to a real file or component in src/; the Orca boxes are sourced from the research docs in this repo.

The two systems converged on the same philosophy from opposite starting points. Both are a thin CLI "ignition key" over a long-lived daemon that supervises agent processes; both treat turn-end as liveness, not completion; both make skills portable and AGENTS.md canonical. They diverge on the hardest problem — how to drive a heterogeneous CLI agent and detect when it is done — and cockpit's recent control-plane + interactive-codex work was explicitly informed by studying Orca.

1 · Cockpit architecture

Cockpit is a multi-agent orchestration layer that runs agent sessions inside the cmux terminal multiplexer, coordinated by an LLM "Command" session at the top and supervised by a launchd-managed daemon (cockpitd) underneath. The orchestration brain is itself an agent session, not hardcoded code.

1a · The agent hierarchy conceptual model

Command → Captain → Crew, plus the autonomous Reactor. Each role is an agent session with a model routed by role; the CLI only bootstraps and wires sessions together.

1c · The control plane — cockpitd daemon src/control/

A launchd-managed daemon that owns agent task lifecycle. State is a pure reducer over events; tasks are typed JSON records on disk; a watchdog detects stalls; an attach protocol streams interactive sessions over a unix socket.

1d · Interactive-codex end-to-end attach renderer in cmux

A codex crew runs as a daemon-owned app-server thread, streamed live into a cmux tab via the attach protocol. Human-in-the-loop questions promote to a Gate if no client is attached. This is the structured-state path cockpit chose over screen-scraping.

2 · Orca architecture

Orca is an Electron desktop IDE that hosts any CLI agent as its native interactive TUI inside a node-pty, rendered with xterm.js, one per git-worktree pane. It is fundamentally a terminal multiplexer for agents, with a PTY daemon, a unified agent-status pipeline, and a polling orchestration coordinator bolted on. (Source: docs/research/2026-05-19-orca-full-system-study.md.)

3 · Side-by-side comparison

Where the two systems map onto each other, and where they deliberately differ. Cockpit's recent control-plane and interactive-codex work was directly informed by the Orca study — the table notes what cockpit adopted versus what it intentionally did differently.

The core philosophical split. Both systems agree the only trustworthy "done" signal is the agent's own runtime event, and both demote screen-scraping to a last resort. They diverge on the mechanism: Orca embraces heterogeneity — it installs each agent's native hook system and normalizes ~10 event vocabularies server-side, accepting an invasive 8-home-dir footprint to get real signals and a human-attachable native TUI. Cockpit chose structured protocol state over scraping — driving interactive codex through the codex app-server JSON-RPC interface, keeping a pure, auditable typed state machine and a non-invasive footprint, at the cost of less multi-agent breadth today.

What cockpit ADOPTED from the Orca study

  • resumeRef persisted on every state transition (codex threadId) → reattach on daemon boot
  • Daemon "ready" = successful app-server initialize round-trip, not just child-spawned (counters codex 0.129+ silent degradation)
  • DispatchAttempt[] sub-records on TaskRecord — the "sling pattern" so retries don't dirty task state
  • The Gate decision-gate HITL primitive (block → structured question → resolution injected)
  • App-server over Orca's own hook path — Orca's codex-fetcher.ts proved a clean JSON-RPC client works
  • Treat resumeRef as an opaque hashed token, never parsed (Orca #1148 session-id-shape lesson)

Where cockpit deliberately DIFFERS

  • Pure reduce(state,event) + typed JSON state vs Orca's opaque serialized-screen "truth"
  • Anti-false-done elevated to a stated invariant, not patched reactively (Orca's #1437 / tui-idle bugs validate this)
  • Non-invasive ~/.config/cockpit footprint vs Orca's 8 agent home dirs
  • Orchestration is an LLM session (decomposes specs) vs Orca's deterministic-but-non-AI coordinator
  • Self-improving subsystems (learnings, LLM-wiki, self-heal) — no Orca analog
  • External cmux substrate vs Orca's in-app Electron + node-pty ownership
