1 · Big Picture

A crew's lifecycle change reaches the captain's live cmux pane through four hops: a LifecycleSource normalizes a raw signal, the daemon reduces it to a ControlEvent and appends a mailbox message, the delivery loop ticks every ~1s reading unacked entries, and the CaptainDelivery gate in cmux.ts only ever types into a positively-confirmed-empty input box.

2 · Three LifecycleSource Adapters, One Reducer

All three implement the same LifecycleSource port (packages/core/src/lifecycle-source.ts) and normalize into LifecycleSnapshot: { taskId, state, alive, origin, at }. Every snapshot feeds reduceLifecycle(prev, next) — a pure function with 4 rules: agent-originated signals are authoritative; a scan can never assert needsInput; agent-set needsInput is sticky against scans; a stale scan never regresses a more-recent agent state.

Wired in packages/cli/src/squadrantd.ts: each source gets its own deps.report() closure with a prevSnaps map keyed by taskId. After reduceLifecycle the daemon diffs old vs. new state and calls ctx.d.handle({kind:"event", event:{type: "task.blocked"|"task.turn.completed"|"task.progress"|"task.session.ended", ...}}) — this is where a normalized lifecycle state becomes a ControlEvent the daemon reducer understands (Slide 3).

3 · Daemon Reduce → Mailbox → Delivery Tick

The daemon's reduce() (state-machine transition) calls deps.notify({project, message, record, event}) on notify-worthy transitions. The default notify is defaultNotify (daemon/delivery-loop.ts), which appends a formatted message to a per-project mailbox file on disk — decoupling event production from captain delivery.

Stale-skip (#332): entries older than sessionStartMs - STALE_THRESHOLD_MS at daemon-boot are silently acked (cursor advanced, no delivery) — except terminal kinds, which always deliver even late. This stops a fresh daemon restart from replaying a captain's entire historical backlog, while guaranteeing a task.blocked or task.done is never silently dropped.

4 · CaptainDelivery Gate — sendToSurface() in cmux.ts

The last hop is the highest-stakes one: never type into the captain's live keyboard input. parseDraftFromScreen returns a 3-state read of the screen; hasModalOptionList (#484) catches AskUserQuestion/permission modals that the 3-state read can't distinguish from an empty or ghost box; the probe (#302) only fires when content has been byte-stable for N polls.

Not part of this gate: hasCCInputBox() (also defined in cmux.ts) is used by a different pipeline — sendFirstTurnWhenReady() in packages/workspaces/src/crew-pane.ts, which gates the captain → newly-spawned-crew first message, not crew → captain delivery. Both share the HR-boundary screen-parsing approach but serve opposite directions of the mailbox flow (see the #466 slow-boot investigation for the first-turn gate's own history).
