2 · The two-gate security model (answers “why does messaging work if remoteControl is off?”)

Every inbound message passes Gate 1. Only privileged actions also require Gate 2. They are independent.

Gate 1 — Chats allowlist basic relay

cfg.chats — list of chat IDs. An inbound update is honored only if chats.includes(chat.id).

Unlocks the two-way pipe:

  • Daemon → your topic (lifecycle notifications)
  • Your message → the captain’s session (captain.message)
  • Captain → you (squadrant telegram send)

✔ This is why you can chat with the captain today — remoteControl plays no part here.

Gate 2 — remoteControl + user-id privileged

isControlEnabled(cfg) = remoteControl===true  AND  isAuthorized(fromId) = users.includes(your-id). Fail-closed — both required.

Gates only the powerful actions:

  • Auto-launch a down captain on inbound
  • General-topic command channel (/status, /spawn, /crews…)
  • /notify, /mute, /unmute toggles

✘ Currently OFF → slash-commands reply “⛔ not authorized”. Turn on with squadrant telegram setup.

Mental model: Gate 1 = “can this chat talk to me.” Gate 2 = “can this person command me.” You’ve only ever passed Gate 1, which is enough to converse — but not to run commands.

3 · Outbound flow — daemon → Telegram

A crew lifecycle event (or a captain message) leaves the daemon and lands in a topic — after passing the notification filters.

 crew event (task.started / task.done / task.blocked …)
        │
        ▼
 pushLifecycle(project, ev)          fire-and-forget; a Telegram error never crashes the daemon
        │
        ▼
 deliverOutbound(project, ev)
        │
        ├─ resolveNotify(global, projects/<name>.json)   built-in → global → per-project (deep merge)
        │       → { active, cap, crew }
        │
        ├─ active?  live state (telegram-state.json) wins over config default
        │       └─ false → DROP  (muted — no topic create, no send)
        │
        ├─ crew tier?  tierIncludes(crew, ev.type)
        │       └─ false → DROP  (e.g. crew=none drops everything; done_only drops task.progress)
        │
        ▼
 lazily createForumTopic if first time → sendMessage(supergroup, threadId, text)   → appears on your phone

 (captain messages — `squadrant telegram send` — take a parallel path gated by cap instead of crew-tier)

4 · Inbound flow — Telegram → daemon

The daemon long-polls getUpdates. Each message is classified by which topic it came from.

 getUpdates(offset, 50s)  ── long-poll loop, sleeps pollMs between rounds, advances offset (at-least-once)
        │
        ▼
 message arrives  ──►  chats.includes(chat.id)?  no → ignore   (Gate 1)
        │ yes
        ▼
 has message_thread_id?
        │
        ├── NO  → # General topic = command channel
        │           └─ remoteControl + user-id? (Gate 2)  no → “⛔ not authorized”
        │                 └─ parseCommand → run curated squadrant … argv → reply output
        │
        └── YES → # project topic
                    ├─ text is /mute//unmute//notify…?
                    │     └─ Gate 2 → toggle state/config, reply ✅   (NOT appended as a message)
                    ├─ else: auto-unmute this project (engagement, independent of Gate 2)
                    └─ append as captain.message → delivered into the captain’s cmux session as a turn

5 · The three notification dimensions

Independent switches, each resolvable globally or per-project. active is the master per-project gate; crew & cap route the two content streams.

DimensionControlsValuesSet viaStored in
activethe whole topic on/off (the #406 mute gate)on / offnotify <p> on|off, /mute /unmute, auto-unmute on inboundstate (telegram-state.json) — live/auto
crewcrew lifecycle eventsall · alert_only · done_only · nonenotify <p> crew <tier>, /notify crew …config (projects/<name>.json) — deliberate
capcaptain’s own messageson / offnotify <p> cap on|off, /notify cap …config (projects/<name>.json) — deliberate

Tiers are cumulative: none ⊂ done_only {done, failed} ⊂ alert_only {+blocked, approval, input, timeout} ⊂ all {+started, progress, idle, …}.

Config vs. state rule: what you deliberately set → config file (global or per-project); what the system tracks at runtime (live mute, auto-unmute) → telegram-state.json. /mute writes state; /notify writes config.

6 · Where everything lives

~/.config/squadrant/config.json → telegram

botToken      bot identity (BotFather)
supergroupId  the forum group
chats[]       Gate 1 allowlist (chat IDs)
users[]       Gate 2 allowlist (user IDs)
remoteControl Gate 2 master switch
pollMs        getUpdates cadence
notify{active,cap,crew}  GLOBAL defaults

Read at daemon boot (cached). Global changes need a restart to affect the live daemon.

~/.config/squadrant/state/telegram-state.json

offset    getUpdates cursor (at-least-once)
topics{}  project → message_thread_id
notify{}  project → live active bool

~/.config/squadrant/projects/<name>.json

telegram.notify{cap,crew}  per-project override

Read fresh per event — per-project changes take effect on the live daemon immediately.

7 · Concept glossary

ConceptWhat it is in squadrant
BotThe BotFather token identity. The sole channel between daemon and Telegram — pushes via HTTP API, pulls via getUpdates.
Forum supergroupA Telegram group with Topics enabled (supergroupId). Hosts one topic per project.
Topic (thread)A forum sub-channel (message_thread_id). Each project gets one, created lazily on first delivery. The General topic (no thread id) is the command channel.
pushLifecycle / deliverOutboundThe outbound hook: crew lifecycle → topic, after the active/crew/cap filters. Fire-and-forget, crash-contained.
getUpdates loopThe inbound long-poll. Classifies each message by topic, advances the offset for at-least-once delivery.
captain.messageAn inbound message in a project topic, injected into that captain’s cmux session as a conversation turn.
Command channel (#402)The General topic. Curated /status /crews /spawn … mapped to safe CLI argv. Gate 2 required.
Auto-launch (#403)On an inbound to a project topic whose captain is down, boot it first (Gate 2 required).
remoteControl (#321)Gate 2 master switch. Off by default → privileged actions denied. Enabled via squadrant telegram setup.
notify gate (#406) + tiers (#407)Per-project active mute + crew/cap dimensions resolved built-in→global→project.
