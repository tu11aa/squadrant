Orca — Whole System & Deep Comparison vs. Cockpit

Every major subsystem with file:line evidence, then a rigorous head-to-head against cockpit's post-PR-#85 architecture.
Date 2026-05-19 Orca @ 03b88951 (~2000 PRs) Cockpit @ develop 9e61220 Companions orca-codex-wrapping · cockpit-vs-orca
What orca actually is An Electron terminal-multiplexer-for-agents. It hosts ~30 CLI agents as their native interactive TUI inside node-pty, one per worktree pane. The agent loop is PTY-centric end to end; protocols (codex app-server) are used only for side-channel telemetry.
How it drives everything — three tiers, degrading:
  1. Primary — native hooks. It installs each agent's own hook system (9 bespoke hook-service.ts files writing into 8 third-party home dirs), and a server-side normalizer collapses ~10 incompatible event vocabularies into 4 canonical states (working/blocked/waiting/done) via a never-guarded exhaustive switch.
  2. Fallback — OSC-title screen-scraping for tui-idle waits — which orca itself documents as buggy (#1437 stuck-spinner; null→idle hangs).
  3. Last resort — terminal output reading, explicitly demoted in its own skill docs.
The single biggest strategic insight for cockpit Issue #86 is not a process-supervision problem — it's a missing resume-artifact problem, and orca proves the fix. Orca survives daemon/app bounce because each session persists a first-class resumable representation written on every state transition (its "final checkpoint" snapshot), plus a versioned protocol and restart-coalescing. Cockpit's headless path doesn't need a screen snapshot — it needs the equivalent: persist an opaque resumeRef (codex-exec rollout id / claude --resume id / app-server thread id) into per-task JSON on every transition, and make state-machine recovery re-attach through it on bounce. Pair with a daemon protocol-version constant + concurrent-restart coalescing. That single change closes #86 properly and is the highest-leverage thing this study surfaces.

Part A — Orca: The Whole System

A0 · One-paragraph architecture

Orca is an Electron desktop IDE that hosts any CLI agent as its native TUI in a node-pty, xterm.js-rendered, one per git-worktree pane. Three layers bolt on: (1) an out-of-process PTY daemon (src/main/daemon/) that owns the pty children and survives Electron restarts via server-side headless-terminal snapshots; (2) a unified agent-status pipeline that installs each agent's own native hooks and normalizes ~10 vocabularies into 4 states over a loopback HTTP server; (3) an RPC runtime (src/main/runtime/) exposing an orca CLI over AF_UNIX (plus WebSocket+E2EE for mobile), on which a polling orchestration coordinator dispatches a SQLite task DAG by typing prompts into PTYs and collecting worker_done callbacks.

A1 · Generic agent-driving model — the crux

The per-agent launch table is src/shared/tui-agent-config.ts — TUI_AGENT_CONFIG for 30 agents (:58–260). Each entry: detectCmd/launchCmd/expectedProcess, promptInjectionMode, optional preflightTrust, draftPasteReadySignal.

  • promptInjectionMode (:3–9): argv (claude, codex, cursor — codex's positional auto-submits so a draft paste is still needed, :75–79); flag-prompt* (opencode --prompt, gemini, copilot -i because --prompt would run non-interactively and kill the hosted TUI, :242–246); stdin-after-start (aider, goose, amp, ~15 others — screen-timing-dependent injection).
  • preflightTrust (:36–43): pre-writes the agent's "trust this folder" artifact (agent-trust-presets.ts) so the menu never eats the bracketed paste.
  • draftPasteReadySignal (:44–49): generic agents use a quiet-render timer; codex gets the stronger codex-composer-prompt (watches the rendered › prompt) — the only place orca reads codex's TUI, and only for paste-timing.

Onboarding a new agent = one TUI_AGENT_CONFIG row + (if it has hooks) a src/main/<agent>/hook-service.ts.

Turn/done normalization across heterogeneous agents — THE key decision

Every agent has a different native hook system. Orca writes a per-agent managed script and normalizes server-side. Nine hook-service.ts files, each on that agent's own vocabulary:

Normalization is server-side in src/shared/agent-hook-listener.ts: per-source normalize<Agent>Event() map every vocabulary onto 4 canonical states working|blocked|waiting|done (agent-status-types.ts:6). An exhaustive switch with a never guard (isNewTurnEvent :1274–1305) forces a compile error if a new source lacks a turn-start mapping — "fail at typecheck, not silently false."

The crux answer — "does every agent use hooks?" NO, three tiers (1) native hooks → 4-state normalization. (2) OSC-title scraping for tui-idle (orca-runtime.ts:1797–1872) — resolves on any →idle because "Claude Code may skip 'working' on fast tasks, going null→idle, and the waiter would hang forever" (:1861–1866); the #1437 stuck-spinner bug is field evidence this is fragile. (3) terminal output read (120-line buffer) — skill doc: "for status monitoring, not result extraction. Prefer structured worker_done."

Non-obvious tradeoff: orca accepts heterogeneity over a protocol — 8 third-party home dirs, 9 installers, a giant switch — to get real turn signals without scraping. Cost: enormous surface area + version coupling (codex 0.129+ trust hashes).

A2 · PTY / process layer

local-pty-provider.ts: node-pty, pane→process by pty id, env scrubbed so child identity ≠ parent (:232–271). ssh-pty-provider.ts mirrors over SSH. Native gotcha (:39): node-pty's NAPI ThreadSafeFunctions for onData/onExit must be disposed before kill or they "leave callbacks alive into Electron shutdown, which can abort the app."

A3 · Hook ingestion & install lifecycle

agent-hooks/server.ts: loopback http.createServer, bearer-token auth, IPC fanout, on-disk last-status.json surviving restart. Shared listener so an SSH relay can host the same pipeline remotely ("relay normalizes; Orca routes"). Managed-script lifecycle: createManagedCommandMatcher matches by filename not exact command so installs sweep stale entries without clobbering user hooks. Scripts re-source $ORCA_AGENT_HOOK_ENDPOINT each fire so a surviving pty keeps reporting to a new orca instance after restart.

Most reusable production gotcha in the repo Codex 0.129+ silently drops untrusted hooks. Orca computes the trusted_hash (mirroring codex-rs fingerprint.rs + hooks/src/lib.rs), writes per-hook trust into ~/.codex/config.toml, and reports partial if trust is stale because "a green status without trust verification is misleading" (config-toml-trust.ts; hook-service.ts:150–152,297–338). General lesson: codex silently degrades when config preconditions are unmet — never treat silence as success.

A4 · Orchestration coordinator

runtime/orchestration/{coordinator,preamble,db}.ts, SQLite-backed:

  • Task DAG: task-create --deps; task→ready when deps complete; completion auto-promotes dependents.
  • Loop (coordinator.ts:148–225): poll tick() every 2 s — messages → escalations → decision-gates → stale-warn → dispatch → convergence. Decomposition is NOT AI — tasks pre-created; AI-decompose explicitly deferred to "a future phase where the coordinator itself is an LLM agent" (:201–206).
  • Dispatch (:480–574): git-drift guard (skip-not-fail if worktree >20 commits behind — burning the breaker would convert recoverable→hard-fail); then runtime.sendTerminal(handle,{text:preamble,enter:true}) — types the prompt into the pty; writeTerminalAction writes text, setTimeout 500ms, then \r ("TUI apps treat a single large write as a paste"). This is exactly cockpit's legacy cmux injection.
  • Feedback: agent voluntarily runs orca orchestration send --type worker_done (taught by preamble.ts); cooperative callback, no protocol.
  • Watchdog: 5-min heartbeat, 10-min hung threshold; warnStaleDispatches only warns, deliberately does NOT auto-fail ("false-positive cost of killing a slow-but-correct worker > false-negative of a hung worker holding a slot"); heartbeats keyed by dispatchId not task, so a stale heartbeat from a dead retry can't mask a hung new dispatch (:274–301).
  • Circuit breaker: 3 consecutive failures → circuit_broken. Dispatch contexts are separate rows from tasks ("sling pattern") so retries don't dirty task state.
  • Decision gates: HITL — block task, human resolves, resolution injected into the next preamble.

A5–A7 · Worktrees · Source control · Integrations

Worktrees: git-worktree-native, one pane per worktree; drift guard treats stale-base as a dispatch concern not a worktree concern. SCM: diffs in main, rendered in renderer; inline commit; commit-message is the only non-agent codex exec use (stdin delivery for diff-size safety). Integrations: GitHub/Linear/GitLab/Gitea/Azure/Bitbucket dirs; SSH relay daemon SCP'd over SSH, framed JSON-RPC, keeps PTYs alive in a grace period on a unix socket across disconnects, hosts the same hook pipeline; mobile React Native over WS + E2EE + pairing + device registry, renders the daemon's headless snapshots.

A8 · The PTY daemon ★ (the standout — direct cockpitd analog)

src/main/daemon/, 59 files. Electron fork()s it (daemon-init.ts:12), AF_UNIX chmod 0o600, token auth, protocol versioning (types.ts:6 PROTOCOL_VERSION=7, PREVIOUS=[1..6] — "daemons survive app updates; bump for wire-shape changes").

  • Server-side headless terminal: each session runs a @xterm/headless Terminal + SerializeAddon — the daemon keeps the rendered screen, not just bytes.
  • Reattach via snapshot: on reconnect, getSnapshot() returns TerminalSnapshot{snapshotAnsi}; a "final checkpoint" snapshot persisted on exit. Sessions whose subprocess hasn't exited are not reattached as new (guard terminal-host.ts:66–76).
  • Restart orchestration: documented 7-step atomic provider-swap; restartInFlight promise coalescing so two restart triggers can't both enter teardown (daemon-init.ts:40–46); stale-daemon detection + killStaleDaemon.

Net: Electron main/renderer can crash and restart; the pty children keep running in the daemon; the UI reattaches to a serialized screen. The agent conversation still isn't truly resumable (it's whatever the live pty holds) — but the process and screen survive. This is orca's answer to the daemon-bounce-orphans class — i.e. cockpit's open #86.

A9–A11 · CLI · Footprint · Self-improvement

CLI as ignition: src/cli/index.ts → RuntimeClient → AF_UNIX/WS RPC; launchOrcaApp() spawns Electron if down — conceptually identical to cockpit's "CLI is an ignition key, not the brain." Footprint: invasive by necessity — writes into 8 third-party agent home dirs; mitigated by filename-scoped matchers + hash-matched cleanup; the opposite of cockpit's ~/.config/cockpit hygiene. Self-improvement: none — telemetry/ is anonymous opt-out analytics, not a feedback loop. No learnings, no wiki, no self-heal. (Cockpit's learnings + LLM-wiki + self-healing daemon have zero orca analog.)

A12 · Real bugs / workarounds / version-gotchas (the gold)

  • Codex 0.129+ silent hook-drop w/o config.toml trust hash — relevant to any codex-hooks path.
  • #1437 stuck-spinner: OSC-title 'working' stays sticky after agent exits → title-scraping for completion is fragile.
  • #1148 opencode regression: sessionId shape changed numeric→<wtId>@@<uuid>; old path-traversal regex rejected every legit id → session-id shape changes are a recurring break source; fixed by SHA-256-hashing the id.
  • node-pty NAPI leak: dispose onData/onExit before kill or Electron aborts on shutdown.
  • Non-interactive flag foot-guns: codex --ephemeral, copilot --prompt kill the hosted TUI; large argv prompt exceeds Win/SSH limits → stdin.
  • Coordinator can't AI-decompose (deferred); allow-stale-base regex matches inside fenced code blocks (accepted v1 limit); tui-idle null→idle race.

B(ii) · The 5 deepest insights for cockpit

  1. Daemon-bounce is solved with a server-side resumable snapshot + reattach + protocol version, not just process supervision. For cockpit's headless path the resumable representation isn't a screen — it's the JSON state + the child's resumable session id. Design the resume artifact as a first-class persisted thing the daemon writes on every transition (orca's "final checkpoint"), not reconstructed at bounce.
  2. No protocol gives turn-done for free across agents. Orca proves the only reliable signal is the agent's own runtime event; titles/output/timeouts are a degrading fallback it openly distrusts (#1437, tui-idle). This is field evidence for cockpit's anti-false-done invariant. The codex fork should bias toward a real codex-emitted event (app-server TurnCompleted as liveness, or codex's own hooks), and treat legacy cmux scrape as the same "last resort" tier orca demotes.
  3. Heterogeneous agents demand a normalization layer; orca's never-guarded per-source switch is the template. If cockpit grows beyond Claude for interactive, it hits orca's exact problem. Grow an explicit normalizeProviderEvent seam with compile-time exhaustiveness feeding reduce().
  4. Orca's watchdog beats cockpit's current heartbeat on three specifics: (a) heartbeats keyed by dispatch-id not task; (b) warn-don't-autofail on stale; (c) circuit breaker as a separate dispatch-context row from the task ("sling pattern") so retries don't dirty task state.
  5. Decision-gates are a clean codeable HITL primitive cockpit lacks. Block task → human answers a structured question → resolution auto-injected into the next dispatch. Exactly cockpit's "semi-automatic, prompted-at-key-moments" ethos, formalized as a first-class gating state.

B(iii) · Where cockpit is genuinely ahead — do NOT regress

  1. Pure reduce(state,event) + typed per-task JSON + TERMINAL_STATES. Orca's live-agent truth is an opaque xterm blob. Keep cockpit's auditable model.
  2. Anti-false-done as a stated invariant. Orca has the bug class and patches reactively; cockpit elevates it to a principle. Don't weaken to match orca's title-scraping.
  3. Non-invasive footprint. Adopt agent-hooks only eyes-open; if so, copy orca's mitigations (filename-scoped matchers, hash-matched cleanup), not the blast radius.
  4. Self-improving subsystems. Zero orca analog. Genuinely differentiating. Keep investing.
  5. Orchestration as an LLM session, not hardcoded code. Orca's coordinator literally can't decompose a spec yet; cockpit's Command/Captain already are that phase. A real lead.

B(iv) · Concrete recommendations (named subsystems/files)

  1. Close #86 with an orca-style resume artifact. In cockpitd's per-task JSON store, persist a resumeRef on every transition: headless codex = rollout id for codex exec resume; claude -p = --resume id; app-server = thread id. Recovery re-attaches via resumeRef. Add a daemon protocol-version constant and coalesce concurrent restarts.
  2. Add normalizeProviderEvent(provider,raw)→CanonicalEvent with compile-time exhaustiveness, feeding reduce(); keep TurnCompleted/Stop as liveness, gated to completion only by an explicit terminal event.
  3. Refactor watchdog to dispatch-attempt granularity — attempts as sub-records with own lastHeartbeatAt + circuitBroken; warn-don't-autofail default; surface to Captain.
  4. Codex-interactive fork: orca's evidence tilts toward app-server, not orca's hook path. Orca only uses codex hooks because it's already hosting the TUI for a human; cockpit's interactive codex has no such forcing requirement, and codex-fetcher.ts:88–261 proves a clean app-server client works. Take the hook-trust gotcha as a general warning: make daemon "ready" contingent on a successful app-server initialize round-trip, not just child-spawned.
  5. Adopt the decision-gate primitive in the Captain layer — first-class gate{pending|resolved|timeout} blocking a DAG node, structured question to the human, resolution injected into the next Crew dispatch.
  6. Steal the foot-gun catalogue into cockpit's runtime drivers: TUI-killing non-interactive flags, argv-vs-stdin for large prompts, node-pty dispose-before-kill (if cockpit ever pty-hosts), and treat resumeRef as an opaque hashed token, never parsed (orca #1148).

What this means for the open brainstorm fork

The study sharpens — and confirms — Path X Recommendation #4 is the decisive one: orca's reason for using codex's hooks instead of app-server is that it already hosts the TUI for a human — a constraint cockpit's interactive codex does not have. Strip that constraint and orca's own working codex-fetcher.ts app-server client is the evidence the protocol path is sound. So Path X (app-server) holds, now with three orca-derived hardening requirements baked in: (a) resumeRef-on-every-transition closes #86 and makes thread/resume real; (b) daemon "ready" = successful initialize handshake, not spawn (counters codex silent-degradation); (c) a normalizeProviderEvent seam so the same machinery generalizes when more interactive providers arrive.

The fork is still formally yours to call (X / Y / Z) — but the full-system study removes the main argument for Y (orca's hook path) by showing why orca chose it doesn't apply to cockpit. Net for the spec: X, with #86's resume-artifact promoted from "Phase-2 nice-to-have" to a Phase-2 cornerstone.
