A deep, evidence-backed study of stablyai/orca and what it means for cockpit's "interactive codex" design.

1 · What Orca Is

Orca ("The AI Orchestrator for 100x builders", onOrca.dev, by stablyai) is an Electron desktop IDE (TypeScript; electron.vite.config.ts, node-pty, @xterm/xterm) that runs any CLI coding agent — Claude Code, Codex, Grok, Gemini, OpenCode, ~30 listed — side-by-side in terminal tabs/panes across git worktrees. It adds worktree management, source-control review, GitHub/Linear integration, SSH remotes, a mobile companion, and an autonomous multi-agent orchestration coordinator.

Architecturally it is a terminal multiplexer for agents, not a protocol client. Its own tagline — "Multi-agent terminals — run multiple AI agents side-by-side in tabs and panes" — is the architecture. Codex is one of ~30 agents and is not special-cased: it flows through the same generic TUI-in-PTY machinery as everything else.

2 · Exactly How Orca Drives Codex

Headline Orca runs codex as its plain interactive TUI inside a node-pty, rendered with xterm.js. It does not use app-server, exec, mcp-server, or remote-control to drive the agent. Turn/done state comes from codex's own native hooks — not the protocol, not TUI scraping.

2.1 — Agent launch is a bare interactive codex in a PTY

src/shared/tui-agent-config.ts:70–82 — the canonical per-agent launch table

codex: {
  detectCmd: 'codex',
  launchCmd: 'codex',          // <-- bare TUI, no subcommand
  expectedProcess: 'codex',
  promptInjectionMode: 'argv',
  preflightTrust: 'codex',
  draftPasteReadySignal: 'codex-composer-prompt'
}

launchCmd: 'codex' — no exec, no app-server. Spawned through node-pty (local-pty-provider.ts:11,317; SSH variant in ssh-pty-provider.ts). Binary resolution is a pure PATH/version-manager probe (codex-cli/command.ts:147 resolveCodexCommand()) — no invocation logic.

2.2 — Turn / "done" detection = codex's native hooks → loopback HTTP

This is the important part. Orca does not infer turn state from PTY output. It installs managed entries into codex's own hooks system and listens for the events.

src/main/codex/hook-service.ts:42–49

const CODEX_EVENTS = [
  'SessionStart', 'UserPromptSubmit', 'PreToolUse',
  'PermissionRequest', 'PostToolUse', 'Stop'
] as const

install() (hook-service.ts:250–340) writes a managed shell script, registers it in ~/.codex/hooks.json per event, and writes per-hook trust entries into ~/.codex/config.toml. The managed script (:103–133) reads the hook JSON on stdin and curls it to a loopback HTTP server with a bearer token:

curl -sS -X POST "http://127.0.0.1:${ORCA_AGENT_HOOK_PORT}/hook/codex" \
  -H "X-Orca-Agent-Hook-Token: ${ORCA_AGENT_HOOK_TOKEN}" \
  --data-urlencode "paneKey=${ORCA_PANE_KEY}" ... \
  --data-urlencode "payload=${payload}"

The receiver is src/main/agent-hooks/server.ts — a loopback http.createServer with bearer auth, IPC fanout, and an on-disk last-status.json cache that survives restart (server.ts:1–12). So:

  • Stop = turn done — a real signal from codex itself, not a heuristic.
  • UserPromptSubmit / PreToolUse / PostToolUse = live in-flight readout (tool name + input preview) — :37–41.
  • PermissionRequest = the human-in-the-loop boundary. Per comment :39–41: the managed script exits without a decision so Codex still shows its normal approval UI, while orca just flips the pane to a red "waiting" state.
Key behavioral fact Codex's interactive approval/question UI is left fully intact; the human answers it directly in the rendered terminal. Orca only observes state transitions out-of-band. It never answers a codex prompt programmatically.

2.3 — The two narrow places codex's non-TUI surfaces ARE used

(a) codex app-server — telemetry side-probe only. src/main/rate-limits/codex-fetcher.ts:84–261. To populate the rate-limit status bar, orca spawns a separate, short-lived codex -s read-only -a untrusted app-server child (:99–106), speaks JSON-RPC over stdin/stdout, asks one question, then kills the child (:191). This is a complete, correct, production app-server client — and it is gold for our Phase 1 (see §7). On any RPC failure it falls back to a PTY that types /status and regex-scrapes the output (:264–461). Telling comment :440: "app-server can fail independently of the interactive CLI."

(b) codex exec — commit messages only. src/shared/commit-message-agent-spec.ts:248–269: codex exec --ephemeral --skip-git-repo-check -s read-only with the diff on stdin. One-shot text generation, not agent orchestration.

(c) Input-timing screen-scrape (minor). draftPasteReadySignal:'codex-composer-prompt' watches the PTY stream for codex's › composer prompt purely to time a bracketed paste — the only place orca reads the rendered TUI to make a decision, and it's not state.

2.4 — Autonomous multi-agent orchestration model

src/main/runtime/orchestration/coordinator.ts + preamble.ts. When orca runs agents autonomously it launches the agent's TUI in a worktree PTY and injects a prompt preamble that teaches the agent orca's own CLI: the agent is told to call orca task complete --body <summary> when done and to heartbeat every 5 min. Completion = the agent voluntarily calling back, backed by a 10-min stale-heartbeat threshold + auto-fail (coordinator.ts:172,230,332–335). Cooperative + watchdog, not protocol-driven. (This is essentially the same pattern as cockpit's legacy captain/crew callback model — and shares its weakness: it trusts the agent to report.)

2.5 — Session / resume model: there is none for codex

Grep for codex.*resume, experimental_resume, rollout, --continue, conversationId in non-test source: nothing. A "session" in orca = a PTY + xterm buffer + a stable paneKey. Resilience across an orca restart = (a) the last-status.json disk cache so dashboard rows reappear, and (b) the managed hook script re-sourcing its endpoint so a surviving PTY keeps reporting. But the codex conversation itself is not resumable — if the PTY dies, the conversation is gone. Orca's resilience story is "keep the PTY alive and re-attach the observer," never "reconstruct the conversation."

3 · Codex Version Traps Orca Encodes

Most reusable production lesson in the repo Codex 0.129+ silently degrades when config preconditions fail. config-toml-trust.ts:14,20; hook-service.ts:151,297: untrusted hooks "sit in the review-required pile" with no error. Orca computes trusted_hash entries to dodge this. Lesson for us: codex can look-alive-but-be-stuck — never treat silence as success.
  • src/cli/codex-command-classification.ts:1–37 enumerates the modern codex surface orca knows (exec, app-server, mcp-server, remote-control, exec-server, stdio-to-uds, cloud…) — used only to classify "safe in background PTY vs. needs a visible terminal," never invoked for orchestration.
  • Model tables list gpt-5.1/5.2/5.3-codex … gpt-5.5 — consistent with a late codex-cli ~0.13x assumption, the same neighborhood as our 0.130.0 target. The 0.129+ hook-trust requirement is directly relevant.

4 · The Wider Landscape ("tons of others wrap codex")

Not corroborated by orca's code or dependencies. Orca depends on no third-party codex wrapper, no @openai/codex programmatic SDK, no codex-pty library. The ~30 names in its README are agents it hosts, not wrappers it uses. Codex's own subcommands (mcp-server, app-server, remote-control) appear only in its command-classifier, never invoked for orchestration. Conclusion: orca isn't evidence of how others wrap codex — it's evidence that one serious, well-engineered multi-agent product independently chose TUI-in-PTY + native hooks over the app-server protocol for the agent loop. (cmux is the same shape: it just runs the codex TUI in a tab — the unreliable part was cockpit screen-scraping that tab, which hooks would fix.)

6 · The Fork — This Is The Decision

Orca doesn't invalidate app-server, but it proves a cheaper path is real and shifts the risk picture. Three options remain genuinely open:

Path X  Stay app-server (Approach 3). Cockpit drives codex over JSON-RPC, renders its own chat. Buys: captain-steerable, resumable across bounce, unified in the control-plane, no global config mutation. Costs: biggest build; the conversation methods (turn/start + approvals) are the one thing nobody — not even orca — has shipped, so Phase 1's smoke test becomes the true risk gate, not a formality.

Path Y  Pivot to orca's model (TUI-PTY + native hooks). Run real codex in the cmux tab; observe turn/done via codex's native hooks → a loopback listener (orca's exact, proven mechanism — reliable, not scraping). Buys: full native UX free, approvals handled by codex's own UI, smallest build, battle-tested. Costs: no captain-steering, no resume-on-bounce (conversation dies with the PTY), and it mutates your global ~/.codex config — a real coordination/cleanliness hazard if your own codex use or multiple cockpit sessions collide.

Path Z  Hybrid — native TUI for the human + app-server thread for state/resume. Cheap full-fidelity UX plus a resumable handle. Open risk: unverified whether a TUI session and an app-server thread can be bound to the same conversation — may be technically impossible; needs a spike before it can be specced.

What changed vs. before the study Before: app-server looked like the only non-scraping path. After: orca proves a second reliable non-scraping path (native hooks) that is far cheaper but sacrifices the two things cockpit's north star most values — resilience (resume) and orchestration (captain steering). The decision is now an explicit values trade, not a technical unknown.

7 · Recommended Path & Adjustments

Recommendation Hold Path X (Approach 3) — but treat orca's findings as mandatory hardening, and run the Phase-1 smoke test as a true go/no-go gate before any daemon work. Rationale: cockpit's stated north star is a reliable orchestrator where one captain controls many crew with auto-recovery. Resume-on-bounce and captain-steerability are exactly that — and they are precisely what Path Y structurally cannot do. Orca's lack of them isn't a counter-argument; it's confirmation those are the differentiating capabilities worth the build.

Phase 1 additions (low cost, high confidence — lifted from orca's working client)

  • Exact handshake ordering: initialize (request, clientInfo) → await response → initialized notification (no id) → only then real methods. Orca comment codex-fetcher.ts:142–146: skipping the notification → "Not initialized" errors. Bake in + smoke-test that it cannot happen.
  • Defensive line framing: newline-delimited JSON, one object per line; ignore non-JSON lines; route id-less messages as notifications (mirror :160–221).
  • Envelope-wrapped results: orca got { rateLimits: {...} }; expect turn/thread results similarly nested. Rely on vendored codex app-server generate-ts types, never hand-written flat shapes.
  • Global flags precede the subcommand: codex -s <sandbox> -a <policy> app-server (orca uses -s read-only -a untrusted; we need writable + an approval policy). Match arg order.

Phase 2 additions (counter the pitfalls orca exposes)

  • "Ready" = successful handshake probe, not just child-spawned — directly counters the codex-0.129+ "silent degradation" pattern. Reinforces our "TurnCompleted is liveness, NOT completion" anti-false-done invariant.
  • Approval round-trip is the highest-risk, zero-prior-art surface. Elevate it to an independent, explicitly smoke-tested success criterion: a turn/start that triggers a tool needing approval → answered over RPC → TurnCompleted. Orca has no reference here — we own this.
  • thread/resume-on-bounce as a tested first-class path. Orca's total lack of conversation resume confirms this is the capability that makes cockpit's interactive codex worth more than orca's — not an afterthought.
  • Supervise the app-server child. Orca keeps a fallback because "app-server can fail independently of the CLI." Daemon must restart it and re-thread/resume live tasks.
Honest residual risk The conversation-driving methods (turn/start, turn/steer, the approval round-trip) are unproven in production by anyone we can find, including orca. Path X is the right bet for cockpit's goals, but Phase 1's smoke test is not a formality — it is the gate that says go/no-go before we invest in the daemon. If Phase 1 reveals the conversation API is unreliable, Path Y becomes the fallback, and that decision is cheap to make at that point because Phase 1 cost little.
