Cockpit · System-Level Comparison

Cockpit vs. Orca

Two multi-agent orchestrators, opposite centers of gravity: a headless control-plane vs. a desktop terminal multiplexer.
Date 2026-05-19 Cockpit @ develop 9e61220 (PR #85 merged) Orca @ 03b88951 Companion orca-codex-wrapping-study.html
In one sentence Orca is a polished desktop GUI that makes a human productive watching many agent TUIs at once; Cockpit is a headless control-plane that makes a captain agent reliably command many crew agents with state, recovery, and no human in the loop. Orca optimizes human-in-the-room ergonomics; cockpit optimizes agent-commands-agents reliability.

1 · The Essential Difference

COCKPIT — agent-first

The CLI is an "ignition key, not the brain." It bootstraps a session; orchestration then happens inside an agent session: Command → Captain → Crew, plus a Reactor. The recent control-plane (PR #85) makes the daemon the spine: tasks are records, transitions are a pure function, liveness is a heartbeat — explicitly so an agent can drive other agents without a human watching.

ORCA — human-first

"Run multiple AI agents side-by-side in tabs and panes." The product is the multiplexer: worktrees, xterm panes, source-control review, Linear/GitHub, a mobile companion. Autonomy exists (the coordinator) but the center of gravity is a human reviewing diffs and answering approvals in real terminals.

This single difference cascades through every other design choice below. Neither is "better" — they optimize different objective functions. Cockpit's bet: the future is agents commanding agents, so invest in state and recovery. Orca's bet: the human stays in the loop for a long time, so invest in seeing and touching the agents.

2 · Architecture

LayerCockpitOrca
RuntimeNode CLI + cockpitd launchd daemon; AF_UNIX socket; source ~/me/claude-cockpit, runtime ~/.config/cockpitElectron main + renderer; node-pty; xterm.js
Process modelDaemon owns headless child PIDs; interactive = injected hook eventsOne PTY per agent pane; agent is the long-lived process
StatePer-task JSON records; pure reduce(state,event); TERMINAL_STATES; anti-false-done invariantPTY + xterm buffer + paneKey; last-status.json disk cache
TransportLocal socket protocol (request/response today)Loopback HTTP hook receiver (bearer auth) + IPC fanout
UINone of its own — borrows cmux tabs; reports to ObsidianFull native desktop UI, panes, diff review, mobile app
The structural mirror Both independently arrived at "a privileged local process owns agent children and collects out-of-band events over a local socket." Cockpit's daemon + AF_UNIX + state machine ≈ orca's Electron-main + loopback-HTTP + status cache. They diverge only on what the event source is (cockpit: provider-native headless JSON / injected hooks; orca: codex-native hook callbacks) and whether a human is the consumer (orca renders it; cockpit feeds a captain).

3 · Agent-Driving Model

COCKPIT

  • Headless: claude -p, opencode run, codex exec --json as daemon-owned children; structured result parsing
  • Interactive (Claude only today): a real chat session in a cmux tab
  • Legacy crew = cmux screen-scrape (spawn/send/read/close) — the unreliable path the control-plane exists to retire
  • Provider tiering = (provider × mode) — generalized, not Claude-only

ORCA

  • Every agent (codex incl.) = its native TUI in a node-pty, no subcommand
  • Turn/done = the agent's native hooks → loopback HTTP (Stop = real done)
  • app-server/exec used only for telemetry/commit text — never to drive
  • Approvals: human answers in the TUI; orca only observes

Sharpest contrast. Cockpit is migrating away from terminal scraping toward structured/protocol signals. Orca never scraped for state in the first place — it leans on the agent's own hook events while still showing the human the real TUI. The cross-study (orca-codex-wrapping-study.html) is precisely about whether cockpit's interactive-codex should adopt orca's hooks model (cheap, proven, human-only) or push the unproven app-server conversation protocol (steerable, resumable). That decision is still open.

4 · Orchestration Model

CockpitOrca
HierarchyCommand → Captain → Crew + Reactor (declarative GitHub polling)Flat panes; optional orchestration coordinator over worktrees
Completion detectionState machine on real events; heartbeat watchdog; mandatory captain close-outAgent voluntarily calls back orca task complete + 5-min heartbeat / 10-min stale auto-fail
Trust modelDon't trust the agent to report — derive state from eventsCooperative — trusts the agent to report, watchdog as backstop
Process disciplineGSD plans, Karpathy principles, TDD, learnings/wiki self-improvementWorktree isolation, source-control review, Linear/GitHub task sync
Notable Orca's autonomous completion model (inject a preamble teaching the agent to call orca task complete, backstopped by stale-heartbeat auto-fail) is structurally identical to cockpit's legacy captain/crew callback model — and shares its weakness: it trusts the agent to self-report. Cockpit's control-plane is a deliberate step beyond this, deriving completion from real provider events rather than agent goodwill. This is the clearest place cockpit is ahead of orca architecturally.

5 · Multi-Provider Support

COCKPIT

  • Driver abstraction + plugin slots (runtime/workspace/tracker/notifier)
  • Claude (reference), Codex, OpenCode shipping; Gemini/Aider in progress
  • Depth over breadth: each provider gets a real headless adapter, sandbox/cwd, state integration
  • Cross-agent projection layer planned (issue #31)

ORCA

  • ~30 agents in a generic launch table — Claude, Codex, Grok, Gemini, OpenCode…
  • Breadth over depth: any agent that has a TUI "just works" because the model is generic PTY + that agent's hooks
  • Codex is not special-cased — same machinery as all others

Orca's generic-TUI model gives near-free breadth (add a row to a table). Cockpit's adapter model costs more per provider but yields structured control, sandboxing, and state the generic model can't. Orca proves breadth is cheap if you accept the TUI+hooks ceiling; cockpit is paying for a higher ceiling.

6 · State, Resilience & Recovery

CockpitOrca
Durable task stateYes — on-disk records, atomic writes, path-traversal hardenedStatus cache only (last-status.json); no task graph
Conversation resumeThe goal (thread/resume in the interactive design); headless orphan-on-restart is open issue #86None — PTY dies, conversation gone
Watchdog / livenessHeartbeat budget per task; stalled ≠ terminalStale-heartbeat auto-fail (cooperative)
Self-healDaemon self-heals every CLI invocation; runtime-syncRe-attach observer to surviving PTY
Honest gapDaemon bounce still orphans in-flight headless (#86)No recovery story for a lost conversation at all
Where cockpit is structurally stronger Resilience is cockpit's whole thesis. It has a durable state machine and an explicit recovery roadmap; orca has neither — its resilience is "keep the process alive." This is exactly why the interactive-codex study recommends holding the app-server path: thread/resume-on-bounce is a capability orca structurally cannot have, and it is core to cockpit's reason to exist.

7 · Human Surface & UX

ORCA — wins decisively here

  • Native desktop UI; many agent TUIs visible at once, full fidelity
  • Source-control review, diff UX, Linear/GitHub integration
  • SSH remotes, a mobile companion app
  • Approvals answered in the agent's own polished UI

COCKPIT — deliberately thin

  • No GUI of its own; rides cmux tabs; Obsidian hub-spoke for reports
  • Semi-automatic: prompts at key moments, not a dashboard to watch
  • Daily briefing, standup/retro, learnings — text/agent-mediated
  • The human is meant to be mostly absent

This is the inverse of §6. Orca is years ahead on human ergonomics and visibility; cockpit intentionally declines that race (its memory even records "context preservation beats visual side-by-side panes"). They're not competing here — they disagree about whether the human should be watching.

8 · Footprint & Invasiveness

CockpitOrca
Global config mutationScoped to ~/.config/cockpit; no global agent-config editsWrites managed entries into global ~/.codex/hooks.json + config.toml trust hashes
Install weightNode CLI + launchd plistElectron app (heavy) but self-contained
Coexistence with your own agent useDesigned to not touch user workspaces/configHook-trust edits can collide with your own codex use / multiple sessions

Cockpit is the less invasive guest on the machine. Orca's power partly comes from rewriting the agent's own config — effective, but a coordination hazard the comparison study flags as a real cost of adopting orca's model.

9 · Extensibility & Self-Improvement

COCKPIT

  • Plugin slots + load-on-demand skills (portable markdown, multi-agent)
  • Self-improving by design: learnings system, LLM wiki compilation, "everything should improve itself"
  • GSD/Task Master integration for complex work
  • AGENTS.md-canonical, Claude-Code-as-reference (not Claude-only)

ORCA

  • Extends by adding agents to a launch table — trivially broad
  • No self-learning/wiki/retro loop evident; improvement is product-team-driven
  • Orchestration via injected preambles teaching agents its CLI

Cockpit treats self-improvement as a first-class subsystem (a stated user principle); orca treats it as a product roadmap. Cockpit's design is more ambitious here and more aligned with an "autonomous, self-recovering" north star — at the cost of complexity orca avoids.

11 · What Each Can Learn From the Other

Cockpit should steal from orca

  • Native-hooks-for-state as a reliable, cheap alternative to a custom protocol — already the open decision in the codex study (Path Y)
  • The exact app-server handshake reference (codex-fetcher.ts:88–261) for Phase 1
  • The codex 0.129+ silent-degradation trap → "ready = handshake probe, not spawn"
  • That breadth is cheap if a generic TUI tier is acceptable as a fallback provider mode

Where cockpit is already ahead

  • Durable state machine & explicit recovery (orca has none)
  • Conversation resume ambition (orca structurally cannot)
  • Deriving completion from events, not agent goodwill
  • Non-invasive footprint (no global agent-config rewrites)
  • Self-improvement as a subsystem

12 · Strategic Takeaways for Cockpit

Verdict Orca is the stronger product (UX, breadth, polish); cockpit is the stronger orchestration substrate (state, recovery, agent-commands-agents). They are not the same category — orca validates cockpit's hard parts by lacking them.
  1. Don't chase orca's UX. A GUI/mobile app is a different product. Cockpit's edge is the headless substrate; doubling down on state/recovery is the right focus, consistent with the stated north star.
  2. Adopt orca's hooks insight as a provider tier, not a pivot. "TUI + native hooks" is a legitimate, cheap, reliable mode. Consider it a fallback interactive tier (especially for the long tail of ~30 agents) under the control-plane — breadth without abandoning the state machine.
  3. The codex-interactive fork stands (see companion study §6). Hold the app-server path for codex specifically — resume + steerability are cockpit's differentiators and orca proves they're rare. Use orca's client as the Phase-1 reference; let the Phase-1 smoke test be the true go/no-go.
  4. Close #86 with intent. Resilience is the entire reason cockpit beats an orca-style model. Orphan-on-restart is the gap between the thesis and the implementation — it's the highest-leverage next investment.
  5. Keep the non-invasive footprint as a deliberate selling point. Orca's global-config rewriting is a real cost; cockpit's restraint is a feature worth protecting.
