# Multi-Agent Orchestration Landscape — How Others Drive Heterogeneous Coding-Agent CLIs

**Date:** 2026-06-25
**Status:** Research synthesis (informs squadrant agent-driver architecture)
**Method:** `deep-research` harness — 107 agents, 5 search angles, 25 sources fetched, 114 claims extracted, 25 adversarially verified (3-vote, need 2/3 to refute), 23 confirmed / 2 refuted.
**Trigger:** Evaluating whether to drop the structured `codex app-server` route + generated TS protocol mirror in favor of driving the plain interactive codex TUI like Claude (to cut perceived bundle bloat).

---

## TL;DR

The ecosystem is moving **away from PTY/screen-scrape, toward agents exposing structured server/event interfaces**. Driving plain-TUI codex would go backwards against both the vendor (OpenAI) and the industry trajectory.

1. **Keep `codex app-server` + the generated mirror.** OpenAI designated app-server the *first-class, officially-maintained* integration path, tried-and-abandoned MCP for the Codex client (rich-session semantics didn't map), and engineered the JSON-RPC surface to be backward-compatible. The 486-file mirror is generated by first-party `codex app-server generate-ts` — vendor-sanctioned, not a fragile workaround. **Regenerate per codex version bump; do NOT delete it.**
2. **Keep the per-agent `AgentDriver` seam.** No mature orchestrator forces one uniform protocol. The closest comparable (Overstory) carries three mechanisms (PTY / headless NDJSON / direct RPC) behind a single `AgentRuntime` interface. *One seam, per-agent mechanism* = exactly squadrant's design.
3. **Do NOT adopt A2A / ACP / MCP as a wire protocol yet** — but borrow their patterns (A2A's 8-state task lifecycle + Agent Card capability model as an internal normalization schema).
4. **The real opportunity:** squadrant's PTY-first Claude/opencode path is the *laggard*, not codex's app-server. The trajectory is structured-first (Overstory defaults Claude Code to headless `stream-json` NDJSON; PTY is its opt-in escape hatch).

## Angle 1 — CLI-agent orchestrators (closest to squadrant)

**Finding (high, 3-0):** Mature orchestrators (Claude Squad, vibe-kanban, Conductor) are **agent-agnostic** — each launches a real coding-agent CLI as a configurable shell command/executor, running many heterogeneous agents (Claude Code, Codex, Gemini CLI, opencode, Aider, Cursor, Amp, Copilot…) interchangeably, with **no per-agent structured API integration**.
- Claude Squad: `tmux new-session -d -s <name> -c <workDir> <program>`; agents configured via `-p` flag or `{name, program}` profile pairs.
- vibe-kanban: pluggable executor enum (claude-code, amp, gemini, codex, opencode, cursor_agent, qwen-code, copilot, droid) — "switch between 10+ coding agents."
- Conductor: bundles parallel Claude Code / Codex / Cursor in isolated workspaces.
- **Implication:** the dominant pattern *is* squadrant's `AgentDriver` seam — agent-agnostic process launch, not a single uniform protocol.

**Finding (high, 3-0):** **Git worktrees** (one branch + working dir per agent) are the converged isolation primitive — chosen over containers — matching squadrant's per-crew worktree model. Confirmed verbatim in Claude Squad, vibe-kanban, Conductor ("every workspace is a separate git worktree, so there is zero conflict").

**Finding (high, 3-0):** The control seam is **shifting from PTY/tmux screen-scrape toward structured event streams.** Overstory (the closest comparable) *inverts* squadrant's PTY-first model — its **default** drives Claude Code headlessly via `-p --output-format stream-json` and parses NDJSON; PTY/tmux is the **opt-in live-steering escape hatch** (`ov sling --no-headless` → `tmux attach`). Caveat: Overstory is now archived (successor "Warren"), but the inversion is code-confirmed.

**Finding (high, 3-0):** Comparable orchestrators do **NOT** converge on a single integration seam — they use a pluggable per-agent runtime interface carrying **multiple** mechanisms simultaneously. Overstory's `AgentRuntime` abstracts ~11–12 runtimes with THREE mechanisms: `buildSpawnCommand` (PTY/tmux), `buildDirectSpawn` + `parseEvents` (headless NDJSON), and optional `connect()` (direct RPC, used by Pi). **Direct precedent for squadrant's structured-codex + PTY-Claude/opencode coexistence.**

## Angle 2 — Agent-interaction protocols & standards

**Finding (high, 3-0):** **MCP standardizes LLM-to-tool/context access** (Tools, Resources, Prompts, Sampling over JSON-RPC) — explicitly **NOT** an agent-to-agent coordination protocol. Wrong fit for captain↔crew transport. (This is *why* OpenAI abandoned MCP for the Codex client driver.) MCP is relevant to squadrant only to expose squadrant's *own* tools to agents.

**Finding (high, 3-0):** **A2A is the standard purpose-built for agent-to-agent task orchestration** — Agent Card capability discovery (`/.well-known/agent-card.json`), a formal **8-state task lifecycle** (SUBMITTED, WORKING, COMPLETED, FAILED, CANCELED, INPUT_REQUIRED, REJECTED, AUTH_REQUIRED; +UNKNOWN sentinel), Task+Artifact messaging over JSON-RPC/HTTP with optional SSE + push. **But** it targets *cross-organizational, opaque-agent* collaboration within trust boundaries — **premature as squadrant's in-process wire protocol.** Its lifecycle/state model is a strong reference for normalizing done/blocked/idle/approval across drivers.

**Finding (high, 3-0):** **ACP (Agent Client Protocol, Apache-2.0) standardizes agent-to-EDITOR integration** (LSP-style; any agent ↔ any editor) — **NOT** agent-to-agent. Gemini CLI is an ACP reference impl; JetBrains partnered Oct 2025. **Refuted (0-3):** the claim of broad 50+-implementation cross-vendor adoption — ACP is real but adoption breadth is unconfirmed; treat as real-but-early, editor-scoped. Matters to squadrant only *if* target agents converge on exposing ACP servers squadrant could drive uniformly (future possibility, not present reality).

## Angle 3 — In-process orchestration patterns

**Finding (high, 3-0):** The **supervisor / handoff patterns** transfer cleanly to squadrant's out-of-process captain→crew model:
- A central supervisor controls all delegation, choosing which agent to invoke by context → mirrors the **captain**.
- Subagents wrapped as opaque tools; only the final response surfaces, intermediate reasoning hidden → validates squadrant **hiding crew internals from the captain, surfacing only DONE/result**.
- Handoffs modeled as explicit tool-calls (`create_handoff_tool` → `Command(goto=agent)`) → maps to `crew signal`/delegation.
- **Critique to heed:** central supervisor = single point of failure + per-routing-LLM-call latency. Squadrant already mitigates: the captain is a *persistent session*, not a per-hop router.

## Angle 4 — Driving heterogeneous CLIs you don't own

**Finding (high, 3-0):** OpenAI has **officially designated the Codex App Server** (bidirectional JSON-RPC over stdio/JSONL) as the first-class, maintained, default-recommended integration method — *after explicitly trying and abandoning an MCP server* because MCP semantics didn't map to rich session interactions (e.g. diff updates). Quote: *"Codex App Server will be the first-class integration method we maintain moving forward… By default, we'd recommend that clients use Codex App Server."*

**Finding (high, 3-0):** The App Server is **engineered for backward compatibility** (older clients ↔ newer server binaries; cf. Xcode decoupling release cycles) and its TS / JSON-Schema types are **officially generated** via first-party `codex app-server generate-ts` / `generate-json-schema`. → squadrant's 486-file mirror is the **vendor-sanctioned path**, not a fragile third-party workaround. Caveats: backward-compat promise lives in the engineering blog (design intent, not contractual); app-server still labeled EXPERIMENTAL; generated artifacts are version-pinned → **regenerate per version bump**, don't hand-maintain.

**Trajectory:** Codex exposes a full structured server *today*. Claude Code has headless `stream-json` (NDJSON) but no full bidirectional server. Gemini CLI is an ACP reference impl. The direction points toward server-exposing agents, but **uniformity is not yet achievable** — a per-agent seam remains correct.

## Architecture recommendations for squadrant

1. **Codex app-server + generated mirror → KEEP both.** Vendor-blessed, version-tolerant. Regenerate on each codex version bump. *Optionally* wire the generated types into `normalize.ts` for real type-safety (currently duck-typed with `any`) — the mirror is **underused**, not dead weight. Repo-hygiene relocation (#429) is fine; **delete-entirely is contraindicated.**
2. **Per-agent `AgentDriver` seam → KEEP.** Industry consensus (Overstory's 3-mechanism `AgentRuntime`). Do not chase a uniform driver.
3. **A2A / ACP / MCP as wire protocol → NOT YET. Borrow patterns instead.** Adopt A2A's 8-state task lifecycle + Agent Card capability model as squadrant's *internal* normalization schema (the shape each driver maps native done/blocked/idle/approval into). Reassess adoption only if/when the four target agents converge on exposing a common server (ACP is the likeliest convergence point).
4. **Investigate the inverse of the original question:** squadrant's *Claude/opencode PTY screen-scrape* is the laggard, not codex's app-server. Evaluate migrating Claude Code PTY → headless `stream-json` NDJSON where lifecycle fidelity matters. **Open question:** does headless stream-json carry enough fidelity (mid-session steering, approval events, interactive back-and-forth) to replace PTY for *long-lived* crews, or only for one-shot crews? → spike candidate.

## Open questions (carried forward)

- Will Claude Code / Gemini CLI / opencode each expose a stable, officially-maintained structured agent-server interface squadrant could target uniformly, and on what timeline?
- Does Claude Code headless `-p --output-format stream-json` provide enough lifecycle/approval fidelity to replace PTY for long-lived interactive crews?
- Is there value in adopting A2A's Agent Card + 8-state lifecycle as the internal driver-normalization schema vs inventing squadrant's own taxonomy?
- At what cost/benefit threshold does adopting an emerging standard stop being premature — and which standard's adoption among the four target agents is the trigger?

## Refuted claims (excluded from findings)

- ACP has broad cross-vendor adoption (50+ implementations across Zed/JetBrains/VS Code/Neovim/Emacs/Obsidian + Claude/Gemini/Cline/Copilot/Cursor/Devin) — **0-3, unconfirmed.**
- All four protocols (MCP/ACP/ANP/A2A) are low-maturity with no production metrics as of early 2026 — **0-3, overgeneralized.**
