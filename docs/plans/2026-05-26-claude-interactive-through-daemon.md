# Cockpit Claude Interactive Through Daemon — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make `cockpit crew spawn --agent claude` daemon-supervised so the captain can learn the true terminal state of a Claude crew without scraping. Mirrors the codex Phase 2 architecture for the interactive-claude slice of #64.

**Architecture:** Single phase, surgical wiring. Five mechanical changes, no new protocol verbs. The control-plane daemon (PR #85) and codex interactive (PR #98) already provide every piece needed; this plan composes them for Claude.

**Tech Stack:** TypeScript (strict, ESM, `target: ES2022`, `.js` import suffix), vitest, commander, node:net AF_UNIX, claude-cli (probed in Task 1 for `--settings` flag support).

**Spec:** `docs/specs/2026-05-26-claude-interactive-through-daemon-design.md`.

**Conventions seen in cockpit (follow these):**
- Tests are vitest siblings in `src/control/__tests__/<source>.test.ts` (control-plane) or `src/commands/__tests__/<source>.test.ts` (CLI). Existing pattern: `src/control/__tests__/interactive-claude.test.ts`.
- All TS imports use `.js` suffix (ESM).
- Pure functions kept pure (`state-machine.ts::reduce`); side effects live in adapters/launchers.
- Anti-#2576 invariant is enforced in `state-machine.ts` *and codified by mapping* in this plan: never map a Claude `Stop`/`SubagentStop`/`SessionEnd` hook to `task.done`.
- Atomic commits: one task = one commit. Commit subject = `feat(claude-iv): <task title>` or `test(claude-iv): …` / `chore(claude-iv): …`.

```typescript
// src/lib/per-crew-settings.ts
import { mkdirSync, writeFileSync } from "node:fs";
import { join } from "node:path";
import { mergeClaudeHooks } from "../control/interactive/claude.js";

export function writePerCrewSettings(o: {
  stateRoot: string;
  project: string;
  taskId: string;
  hookCmd?: string;
}): string {
  const dir = join(o.stateRoot, o.project, o.taskId);
  mkdirSync(dir, { recursive: true });
  const path = join(dir, "settings.json");
  const merged = mergeClaudeHooks({}, o.hookCmd ?? "cockpit crew _hook");
  writeFileSync(path, JSON.stringify(merged, null, 2));
  return path;
}
```

```typescript
export function mapClaudeHookToEvent(
  event: string,
  _payload: unknown,
  taskId: string,
): ControlEvent | null {
  switch (event) {
    case "Stop":
    case "SubagentStop":
    case "SessionEnd":
      return { type: "task.progress", id: taskId, note: event.toLowerCase() };
    default:
      return null;
  }
}
```

```typescript
.action(async (event: string) => {
  const taskId = process.env.COCKPIT_CREW_TASK_ID;
  const project = process.env.COCKPIT_CREW_PROJECT;
  if (!taskId || !project) { process.exit(0); }   // silent: defensive
  let stdin = "";
  for await (const chunk of process.stdin) stdin += chunk;
  let payload: unknown = undefined;
  try { payload = stdin ? JSON.parse(stdin) : undefined; } catch { /* ignore malformed */ }

  }
  process.exit(0);
});
```

# TASK 4 — `cockpit crew signal` subcommand

**Goal:** The crew's explicit done/blocked/failed signal. Reads env, POSTs the right `ControlEvent`. Idempotent on terminal states.

**Files:** `src/commands/crew-control.ts` (add subcommand), `src/commands/__tests__/crew-signal.test.ts` (create).

- [ ] **Step 4.1: Write the test (TDD)**

Spec the *request builder* `buildSignalRequest(event: "done"|"blocked"|"failed", o: {message?, question?, error?}): {kind:"event", project, event:ControlEvent}`. Pure function — tested without daemon.

Cases:
- `done` + message → `task.done` with `resultRef` set to a `_results/<id>.txt` path; message text passed to a write helper (writer mocked in tests).
- `blocked` + question → `task.blocked` with `reason:"crew signaled blocked"` and the question.
- `failed` + error → `task.failed` with the error string.
- Missing env (`COCKPIT_CREW_TASK_ID`) → throws an `Error("not running under a crew …")`.

- [ ] **Step 4.2: Implement the builder**

```typescript
// in src/commands/crew-control.ts (top-level)
export function buildSignalRequest(
  signal: "done" | "blocked" | "failed",
  o: { message?: string; question?: string; error?: string; writeResult?: (id: string, payload: string) => string },
): { kind: "event"; project: string; event: ControlEvent } {
  const taskId = process.env.COCKPIT_CREW_TASK_ID;
  const project = process.env.COCKPIT_CREW_PROJECT;
  if (!taskId || !project)
    throw new Error("not running under a crew (COCKPIT_CREW_TASK_ID/COCKPIT_CREW_PROJECT unset)");
  let event: ControlEvent;
  if (signal === "done") {
    const resultRef = o.writeResult ? o.writeResult(taskId, o.message ?? "") : "";
    event = { type: "task.done", id: taskId, resultRef };
  } else if (signal === "blocked") {
    event = { type: "task.blocked", id: taskId, reason: "crew signaled blocked", question: o.question ?? "" };
  } else {
    event = { type: "task.failed", id: taskId, error: o.error ?? "crew signaled failed" };
  }
  return { kind: "event", project, event };
}
```

- [ ] **Step 4.3: Wire as a subcommand**

In `addControlPlaneCrewCommands`:

```typescript
crew
  .command("signal <state>")
  .description("Emit explicit crew terminal signal (done|blocked|failed). Reads COCKPIT_CREW_* env.")
  .option("--message <m>", "Optional summary written to resultRef (done)")
  .option("--question <q>", "Question to surface to captain (blocked)")
  .option("--error <e>", "Error message (failed)")
  .action(async (state: string, opts: { message?: string; question?: string; error?: string }) => {
    if (state !== "done" && state !== "blocked" && state !== "failed") {
      process.stderr.write(`unknown signal '${state}' (expected: done|blocked|failed)\n`);
      process.exit(2);

      await cockpitdCall(req);
      process.exit(0);
    } catch (e) {
      process.stderr.write(`${(e as Error).message}\n`);
      process.exit(1);
    }
  });
```

## Rollback plan

Each task is one commit. If a task breaks captain workflows in field test, `git revert <sha>` of that single commit restores prior behavior — the new pieces are additive (the dispatch path is new, but the existing `cockpit crew read` / `crew send` / `crew close` cmux verbs are untouched throughout). A full feature revert is `git revert <T6-sha>` which removes the dispatch wiring; remaining commits (helpers, signal verb, hook bridge) are dormant without that wiring.
